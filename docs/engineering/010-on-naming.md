# On Naming

**The right name is the one that needs no explanation.**

---

## The Problem With Clever

Ghostwire was 9 characters.

It required knowing what a ghost wire was — or accepting the metaphor on faith. Ghosts are ethereal. Wires conduct. Together, they suggest something that moves unseen through systems. It's evocative. It's also opaque.

Anyone encountering the name for the first time has to learn it. They have to remember it. They have to explain it to others.

Nine characters of explanation, every time.

---

## What a Name Does

A name isn't just a label. It's a contract.

When you call something "planner," users know what it does. When you call something "oracle-strategy," users have to learn the domain vocabulary first.

The same applies to project names. "Jinn" means magical helper — in Arabic, in English, in most languages that have the word. It doesn't require knowing a reference. It doesn't require accepting a metaphor. It means what it means.

The difference isn't semantic. It's cognitive. Every name that requires explanation is a name that users carry incompletely. They remember part of it. They get it wrong. They have to look it up.

Names that explain themselves don't have this problem.

---

## The Cost of a Bad Name

A bad name has a cost that accrues over time:

**First use**: The user encounters the name and doesn't know what it means. They search for it. They read the documentation. They learn.

**Second use**: The user remembers part of it. They guess. Sometimes they're right.

**Third use**: The user has to explain it to someone else. They say "it's called ghostwire, like the thing from—" and they stop. They don't know how to finish the sentence.

**Nth use**: The user has internalized the name, but they still can't explain it easily. They've learned to live with the gap between knowing and understanding.

This cost is invisible. Nobody tracks "how many times did users have to ask what X means?" The cost is paid in small moments of friction, in documentation that has to explain the name before it can explain the product, in branding that has to fight the name instead of amplifying it.

---

## The Character Count Problem

Nine characters vs. four characters.

This sounds trivial. It's not.

Every command invocation is nine extra characters of typing. Every tab completion has nine characters to disambiguate. Every documentation example has nine characters that could be spent explaining something useful.

The CLI is where names live. Users type them thousands of times. Four characters is easy. Nine characters is a tax.

The tax is small on each keystroke. It compounds.

---

## The Meaning Test

Before naming anything — a project, a command, a variable, a concept — apply the meaning test:

**Can someone who has never seen this before understand what it is from the name alone?**

If the answer is no, the name is doing something other than naming. It's being clever. It's being evocative. It's being memorable.

These aren't bad things. But they're different from naming. If you want a name that names, choose a name that means something.

---

## What We Chose

We chose Jinn.

Not because it's shorter — though it is. Not because it's memorable — though it is. Because it means something to anyone who encounters it, in any language.

A jinn is a magical being that grants wishes. In the context of an AI assistant, it suggests: something intelligent, something helpful, something that does what you ask.

The name teaches. It doesn't require a footnote.

---

## The Exception

There's a place for names that don't explain themselves: brand names.

"Apple" doesn't explain a computer company. "Windows" doesn't explain an operating system. "Google" doesn't explain a search engine.

These names work because the products became famous enough to own the names. The names became meaningful through association, not definition.

This takes time. It takes success. It takes marketing that most projects don't have.

Before choosing an opaque brand name, ask: can we afford to spend the years and money to make this name mean something?

Usually, the answer is no. The meaning test is cheaper.
